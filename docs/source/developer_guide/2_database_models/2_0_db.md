# Database

## Relational schema with graph semantics

Biofilter 4 uses a relational database as its persistence layer, but the schema
is deliberately designed to support **graph-like semantics**.

At the core of the schema are:

- **Entities**, representing biological identities
- **Entity relationships**, representing typed biological connections
- **Provenance metadata**, capturing where and how knowledge was integrated

This core behaves conceptually like a **knowledge graph**, while leveraging the
strengths of relational systems: performance, transactional safety,
auditability, and mature tooling.

Surrounding the entity core are **domain-specific master tables** that attach
rich attributes to entities without duplicating identity or relationship logic.

---

## Tables vs models

In Biofilter 4, it is important to distinguish between **tables** and **models**:

- A **table** is the physical structure stored in the database.
- A **model** is the SQLAlchemy-based Python representation of that table,
  exposed through the Biofilter API.

Developers interact with **models**, not raw tables. Models define fields,
relationships, constraints, and metadata, enabling composable and safe query
construction while remaining faithful to the underlying schema.

In short:

- **Tables store data**
- **Models define how the data is accessed and manipulated**

---

## Database access and the `Database` class

Biofilter 4 centralizes all database connectivity through a dedicated
`Database` class.

This class is responsible for:

- managing the SQLAlchemy engine and sessions,
- abstracting differences between database backends (SQLite vs PostgreSQL),
- applying connection-level configuration,
- providing a stable integration point between the `Biofilter` object and the
  persistence layer.

By encapsulating database logic in a single component, Biofilter ensures that
ETL pipelines, queries, and reports remain backend-agnostic and consistent
across environments.

---

## Code structure: models and seeds

All database-related code lives under a dedicated `db/` package, which is
intentionally divided into two main areas:

- `db/models/`
- `db/seed/`

This separation reflects the distinction between **schema definition** and
**initial system metadata**.

---

## Model organization by domain

The `db/models/` folder contains SQLAlchemy models grouped by functional or
biological domain. Each `model_*.py` file encapsulates a coherent set of tables
and relationships.

At a high level:

### Entity core

- `model_entities.py`  
  Defines entities, aliases, entity groups, and typed relationships.

### ETL & provenance

- `model_etl.py`  
  Defines source systems, data sources, ETL packages, execution status, and
  metrics.

### Domain master data

- `model_genes.py`
- `model_variants.py`
- `model_proteins.py`
- `model_pathways.py`
- `model_diseases.py`
- `model_go.py`
- `model_chemicals.py`

Each domain file defines master tables and helper structures specific to that
biological area, always linked back to the entity layer.

### System and curation

- `model_config.py`  
  System-level configuration models.

- `model_curation.py`  
  Conflict tracking, curation state, and resolution metadata.

This organization allows developers to extend Biofilter by adding new domains
or models without modifying the entity core.

---

## Modeling principles for developers

When adding or modifying models, developers should follow these principles:

- Do **not** duplicate biological identities across domain tables.
- Always link domain records to entities.
- Store identifiers and synonyms as aliases, not as separate domain records.
- Model relationships **explicitly**, not via implicit joins.
- Keep cross-domain logic out of domain tables.

> The entity layer defines **integration**; domains define **depth**.

---

## Seed data and cold-start initialization

Biofilter databases are initialized using **seed data**, stored as JSON files
under `db/seed/`.

Seed data does **not** represent curated biological knowledge. Instead, it
defines foundational system metadata, including:

- entity groups (e.g. Genes, Variants, Diseases),
- relationship types and semantics,
- source systems and data sources,
- genome assemblies,
- system-level configuration defaults,
- ETL process definitions and status metadata.

Seed data serves two main purposes:

1. **Cold-start initialization** of a new database.
2. **Controlled introduction** of new foundational metadata over time (e.g.
   adding a new source system).

This approach allows Biofilter to evolve without forcing users to rebuild
databases from scratch.

---

## Provenance as part of the schema

All biological data in Biofilter 4 is contextualized by provenance metadata
captured through ETL models.

From a schema perspective, provenance acts as a **control plane** that:

- tracks where data originated,
- records when and how it was ingested,
- supports incremental updates,
- enables auditing and reproducibility.

Developers should treat provenance tables as **first-class components** of the
schema, not optional metadata.

---

## Schema evolution with Alembic

Biofilter 4 uses **Alembic** to manage schema evolution over time.

This allows:

- adding new tables and columns,
- extending existing models,
- evolving ETL metadata structures,
- upgrading long-lived databases safely.

Biofilter follows conservative migration principles:

- prefer additive changes,
- avoid destructive operations when possible,
- preserve entity identifiers and relationship semantics,
- keep migrations reproducible across supported backends.

Schema changes that affect the entity core or relationship semantics should be
carefully reviewed.

---

## Indexing and performance (developer perspective)

Indexes are critical to Biofilter’s performance and are managed centrally where
possible.

Key points:

- entity identifiers and relationships are heavily indexed,
- large domains (e.g. variants) may use partitioning strategies,
- developers may add domain-specific indexes when justified.

Indexes should reflect real access patterns and remain portable across
environments.

---

## Architectural takeaway

The Biofilter 4 database schema is designed to answer a core question:

> How can biological knowledge be stored in a way that is persistent,
> integrated, extensible, and auditable?

- **Entities** provide identity  
- **Domains** provide attributes  
- **Relationships** provide meaning  
- **ETL metadata** provides trust
